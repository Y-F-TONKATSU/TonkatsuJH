Adobe Animate と CreateJS を実際のサイト制作に使う

遅ればせながら、
Adobe Animate と CreateJS を
実際のサイト制作で
使ってみているので、
扱う上でのポイントを
メモしていく。

初回は Ticker と
アニメーションループについて書く。

タイマの集中管理

JavaScript で
アニメーションを扱う際の
定石の一つに、
タイマの集中管理がある。

『JavaScript Ninja の極意』に
「8.4 タイマの集中管理」
という節があり、
詳しい説明がある。
一つのページに一つのタイマ、
という原則を
パフォーマンスと柔軟性の面から
説いている。

CreateJS もこの原則に従い、
Ticker というオブジェクトで
タイマを集中管理する。

Ticker の基本的な使い方

まずは、Adobe Animate のことは
しばらく忘れて、
素の JavaScript に
タイマの集中管理のためだけに
CreateJS を導入することを
考えてみる。
(実際そういう使い方も
できることはできる)

Ticker は createjs 名前空間内に
Singleton 風味に定義されていて、
new せずに使う。
↓こんな感じ。 

var numTick = 0;

//tick イベントのインターバルを 24回 / 秒 に設定
createjs.Ticker.framerate = 24;

createjs.Ticker.addEventListener('tick', function(){

	//tick イベントが何回実行されたかをログ出力
	console.log(numTick + ' 回目の tick');
	
	numTick++;
	
});

framerate プロパティで
フレームレートを設定できる。

addEventListener メソッドで
tick イベントに
リスナー関数を登録すると、
指定したフレームレートで
リスナー関数がループ実行される。

なお、addEventListener で
tick イベントのリスナーを
初めて追加すると、
createjs.Tikcker.init() が
内部で勝手に呼び出されて、
ループが回り始める。
なので、明示的に
ループを開始する必要はない。

timingMode

tick イベントは、
内部では setTimeout または
requestAnimationFrame によって、
送出される。
どちらを使うかは
timingMode プロパティによって
決まる。

timingMode プロパティは
以下の3つの定数値を取る。

  • TIMEOUT (デフォルト)
  • RAF_SYNCHED
  • RAF

TIMEOUT の場合は
setTimeout が、
RAF_SYNCHED と RAF の場合は
requestAnimationFrame が
使われる。

以下、違いを理解するために、
JavaScript での
アニメーションプログラミングの
基本を確認する。

setTimeout

JavaScript で
アニメーションループを回すには
setTimeout/setInterval と
requestAnimationFrame の
2つの選択肢がある。

まずは setTimeout の例。 

var n = 0;

var animate = function(){

	//animate 関数が何回実行されたかをログ出力
	n++;
	console.log(n + ' 回目の実行');
	
	//インターバルを 24回 / 秒 に設定
	setTimeout(animate,  1000 / 24);
	
};

setTimeout は
タイミングが重要な
アニメーションなどを
想定した API ではないので、
アニメーションプログラムでは
推奨されない。
(実際使ってみると、
実用の範囲内だったりはするのだが)

上記の例では
animate 関数が実行された
回数を表示している。
FPS が 24 なので、
Ticker は 1 秒に 24 回
tick イベントを送出する
…のであれば話は早いのだが、
実際は CPU 負荷等の状況によって、
ブラウザは関数の実行を
遅らせることができる。

なので、
「関数が実行された回数」
を基準にアニメーションを
再生すると、
アニメーションのタイミングは
保証されない。
この辺も
『JavaScript Ninja の極意』の
8 章の説明が詳しくて
参考になる。

例えば、
下記のようなコードを書くと、
表示される経過時間と
実際の経過時間は
どんどんずれていく。 

var currentFrame = 0;
var FRAME_RATE = 24;

var animate = function(){

	//animate 関数の実行回数をフレームレートで割って経過時間を算出
	var currentSec = Math.floor(currentFrame / FRAME_RATE);
	console.log(currentSec + ' 秒経過!');
	
	currentFrame++;
	
	setTimeout(animate,  1000 / FRAME_RATE);
	
};

1 秒後に
'1 秒経過!',
2 秒後に
'2 秒経過!',
..
と表示されそうだが、
実際は、
ブラウザが関数の実行を
遅らせるたびに、
表示される経過時間と
実際の経過時間は
どんどんずれていく。

requestAnimationFrame

そこで、推奨されるのが
requestAnimationFrame である。
(requestAnimationFrame の仕様は
Timing control for script-based animations
 を参照)

requestAnimationFrame では、
フレームレートの指定は行わず、
実行するタイミングについては
完全にブラウザ任せになる。
最適なフレームレートは
実行環境で変わるんだから、
実行時に決めるしかないだろう、
といったところだろうか。

なので、
アニメーションを再生する関数は
タイミングがわからない前提で
書く必要がある。
具体的には、
現在の関数の実行までに
経過した時間を
自分で取得する。
↓こんな感じ。 

var startTime = window.performance.now();

var animate = function(){
	
	//経過時間を算出
	var currentTime = window.performance.now() - startTime;
	
	//経過時間を秒単位で算出してログ出力
	var currentSec = Math.floor(currentTime / 1000);
	console.log(currentSec + ' 秒経過!');
	
	requestAnimationFrame(animate);
	
};

requestAnimationFrame(animate);

これで、ある程度正確に
時間の経過を表示できる。

実は setTimeout でも、
上記の例のように
自分で経過時間を計算すれば、
アニメーション再生のタイミングは
大きくずれなかったりもする。

requestAnimationFrame の利点は、
環境によって変化する
関数を実行する最適なタイミングを
ブラウザが実行時に決めてくれる、
という点と、
関数の実行方法の効率が良い、
という点あたりになる。

RAF_SYNCHED

timingMode の話に戻る。
timingMode プロパティに
指定できる TIMEOUT と RAF は
それぞれ
setTimeout と
requestAnimationFrame に
対応している。

なので、RAF を指定した場合は
自分で経過時間を計算する必要がある。
(framerate プロパティの
設定は無視される。)
ただし、
前回の tick イベントから
今回の tick イベントまでに
経過した時間(デルタ時間)が
tick イベントの
イベントオブジェクトの
delta プロパティに
ミリ秒単位で格納されているので、
多少楽に書ける。
↓こんな感じ。 

var currentFrame = 0;

//タイミングモードを RAF に変更
createjs.Ticker.timingMode = createjs.Ticker.RAF;

createjs.Ticker.addEventListener('tick', function(e){

	//経過時間を計算
	currentFrame += e.delta;
	
	//経過時間を秒単位でログ出力
	var sec = Math.floor(currentFrame * 0.001);
	console.log(sec + ' 秒経過!');
	
});

では RAF_SYNCHED は
何なのかというと、
requestAnimationFrame を使いつつ、
できるだけ framerate プロパティの
設定も尊重してくれる。
RAF の利点
(実行効率の良さ)と
TIMEOUT の利点
(デルタ時間を気にせずにかける楽さ)
のいいとこ取りといえる。

なら、これがデフォルトでいいんじゃ…
と思うのだが、
マニュアルによると
TIMEOUT よりも、
フレームレートとのズレが
大きくなる傾向があるようで、
デフォルトは TIMEOUT になっている。

筆者が試してみた感じ、
ほとんど差はなさそうではあった。

requestAnimationFrame のポリフィル

requestAnimationFrame は
比較的新しい機能で、
全てのブラウザで
サポートされている保証がない。
サポートされていない場合、
setTimeout や setInterval に
フォールバックする必要がある。
『プログラミング HTML5 Canvas』の
5章の解説が詳しい。

これも CreateJS では
Ticker がやってくれる。
このへん のコードを参照。

removeEventListener

リスナーを削除するには
removeEventListener に
add したときと
同じ関数への参照を渡せばよい。

例として、
コンストラクタに関数と
PLAY ボタンと
STOP ボタンを渡すと、
簡易なプレイヤーを作ってくれる
関数を考えてみる。 

var makePlayer = function(action, playButton, stopButton){
	
	$(playButton).click(function(){
		createjs.Ticker.addEventListener('tick', action);
	});
	
	$(stopButton).click(function(){
		createjs.Ticker.removeEventListener('tick', action);
	});
	
};

makePlayer(function(){
	console.log('playing');
}, $('#playButton'), $('#stopButton'));

handleEvent

実は tick イベントの
イベントリスナーには
関数だけではなく、
オブジェクトも指定できる。
その場合、そのオブジェクトの
handleEvent メソッドが、
tick イベントが送出されるたびに
実行される。
↓こんな感じ。 

				
var o = {
	'handleEvent': function(e){
		console.log('tick されてる！');
	}
};

createjs.Ticker.addEventListener('tick', o);

Stage 等のクラスも
handleEvent メソッドを持っており、
Adobe Animate が出力するコードは
この仕組みを使って、
ムービーを再生する。
